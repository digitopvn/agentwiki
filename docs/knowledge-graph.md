# Knowledge Graph

Dual-layer knowledge graph enabling typed document relationships, semantic discovery, and AI-powered graph reasoning.

## Dual-Layer Design

### Layer 1: Explicit Edges (Typed Wikilinks)

Documents are connected through wikilinks in markdown content. Each link carries a **typed relationship**:

| Edge Type | Meaning | Example |
|-----------|---------|---------|
| `relates-to` | General relationship (default) | `[[API Design]]` |
| `depends-on` | Source requires target | `[[Auth Module\|type:depends-on]]` |
| `extends` | Source builds upon target | `[[Base Config\|type:extends]]` |
| `references` | Source cites target | `[[RFC 7519\|type:references]]` |
| `contradicts` | Source conflicts with target | `[[Old Approach\|type:contradicts]]` |
| `implements` | Source implements target concept | `[[Design Spec\|type:implements]]` |

**Wikilink syntax** follows standard wiki convention:

```markdown
[[Target Page]]                          → simple link
[[Target Page|Display Text]]             → link with custom display text
[[Target Page|type:depends-on]]          → typed link
[[Target Page|Display Text|type:extends]] → typed link with display text
```

**Storage**: `document_links` table in D1 with columns: `source_doc_id`, `target_doc_id`, `type`, `weight`, `inferred`, `context`.

### Layer 2: Implicit Edges (Semantic Similarity)

Documents with similar content are automatically connected via vector embeddings:

```
Document A ──embed──▶ Vectorize ◀──embed── Document B
    │                    │                      │
    └────── cosine similarity score ────────────┘
                   (0.0 → 1.0)
```

- **Embeddings**: Generated by Workers AI (`bge-base-en-v1.5`) on document creation/update
- **Similarity query**: Vectorize `query()` returns top-K nearest neighbors
- **Cache**: `document_similarities` table stores top-5 results per document (threshold ≥ 0.7)
- **Refresh**: Async Queue job `compute-similarities` re-computes on document changes

## API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/api/graph` | Full graph (nodes + typed edges), filter by category/tag/type |
| `GET` | `/api/graph/neighbors/:id` | N-hop neighbors (1-3 depth), direction filter |
| `GET` | `/api/graph/subgraph/:id` | Ego-network centered on a document |
| `GET` | `/api/graph/path/:from/:to` | Shortest path between two documents (BFS) |
| `GET` | `/api/graph/stats` | Graph statistics (density, degree distribution, orphans) |
| `GET` | `/api/graph/similar/:id` | On-demand Vectorize similarity query |
| `GET` | `/api/graph/suggest-links/:id` | AI-suggested missing links |

**Common query parameters:**
- `types` — Comma-separated edge types to filter: `?types=depends-on,extends`
- `include_implicit` — Merge similarity edges: `?include_implicit=true`
- `depth` — Traversal depth (1-3): `?depth=2`
- `max_nodes` — Limit result size: `?max_nodes=100`

## MCP Tools (AI Agent Integration)

7 tools expose graph reasoning to AI agents:

| Tool | Purpose | Use Case |
|------|---------|----------|
| `graph_get` | Full graph with type filters | "Show me all dependency relationships" |
| `graph_traverse` | Multi-hop traversal | "What does Auth Module depend on?" |
| `graph_find_path` | Shortest path | "How are Design Spec and API connected?" |
| `graph_suggest_links` | Link recommendations | "What docs should link to this one?" |
| `graph_explain_connection` | Relationship explanation | "Why are these two docs related?" |
| `graph_stats` | Graph analytics | "How healthy is our knowledge base?" |

**Example agent interaction:**
```
User: What documents depend on the Auth Module?
Agent: [calls graph_traverse with edgeTypes=["depends-on"], direction="inbound"]
→ Found 4 documents that depend on Auth Module:
  - Login Page (depends-on, 1 hop)
  - API Gateway (depends-on, 1 hop)
  - User Settings (depends-on → Login Page → Auth Module, 2 hops)
  - Admin Dashboard (depends-on, 1 hop)
```

## Database Schema

```sql
-- Explicit edges (typed wikilinks)
CREATE TABLE document_links (
  id            TEXT PRIMARY KEY,
  source_doc_id TEXT NOT NULL REFERENCES documents(id),
  target_doc_id TEXT NOT NULL REFERENCES documents(id),
  context       TEXT,           -- surrounding text for preview
  type          TEXT NOT NULL DEFAULT 'relates-to',
  weight        REAL DEFAULT 1.0,
  inferred      INTEGER DEFAULT 0,  -- 0=explicit, 1=ai-inferred
  created_at    INTEGER NOT NULL
);

-- Implicit edges (semantic similarity cache)
CREATE TABLE document_similarities (
  id            TEXT PRIMARY KEY,
  source_doc_id TEXT NOT NULL REFERENCES documents(id),
  target_doc_id TEXT NOT NULL REFERENCES documents(id),
  score         REAL NOT NULL,  -- cosine similarity 0-1
  computed_at   INTEGER NOT NULL,
  UNIQUE(source_doc_id, target_doc_id)
);
```

## Performance Characteristics

| Operation | Target | Approach |
|-----------|--------|----------|
| Full graph render | < 1s for 1K nodes | Cytoscape.js with `cose` layout |
| Neighbor query (2-hop) | < 200ms | In-memory BFS on tenant graph |
| Shortest path | < 200ms for 5K docs | BFS with adjacency list |
| Similarity query | < 500ms | Vectorize `query()` + D1 metadata |
| Edge type inference | ~2s per link | Workers AI Llama 3.1 8B, batched (10/job) |

## Known Limitations

- BFS loads full tenant graph into memory (acceptable for < 1K docs per tenant)
- Similarity threshold hardcoded at 0.7 (configurable in future)
- Path finding capped at 10 hops maximum
- Edge type inference batched at 10 links per queue job (Workers AI quota)
- Graph visualization optimized for < 10K nodes
